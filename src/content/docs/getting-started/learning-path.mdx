# Recommended order

If you are new to NestForge, follow the docs in this order:

1. [Installation](/getting-started/installation/)
2. [Quick Start](/getting-started/quick-start/)
3. [Feature Modules](/getting-started/feature-modules/)
4. [DTOs, Services, and Routes](/getting-started/dtos-services-routes/)
5. [Build Your First Feature](/getting-started/build-your-first-feature/)
6. [Database Setup](/getting-started/database-setup/) if the app needs SQL persistence
7. one workflow that matches your runtime:
   `Testing`, `GraphQL`, `gRPC`, `WebSockets`, `Microservices`, `Auth/OpenAPI`, `Scheduling`, or `Caching`
8. [Database Workflow](/guides/database-workflow/) when moving beyond `ResourceService`
9. examples
10. concepts and reference pages later

# Why this order works

New users usually get lost when they read the framework in reference order. NestForge is
easier to understand when you first learn:

- how a feature is structured
- how the module wires it into the app
- how a transport is bootstrapped
- how to test the result

After that, the concept and reference pages become much more useful.

# When to read the reference pages

Read the reference pages after you already have a working mental model. They are best
for:

- checking syntax
- understanding available APIs
- comparing options
- debugging specific framework behavior

They are not the best first stop for onboarding.
