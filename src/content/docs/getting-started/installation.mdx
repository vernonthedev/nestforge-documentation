# Prerequisites

To work with the current NestForge workspace you need:

- a stable Rust toolchain
- `cargo`
- `git`

Depending on which example or feature you use, you may also need:

- `protoc` for gRPC code generation in gRPC-first projects
- a reachable database if you are testing database-backed application code

# Clone and validate the workspace

From a clean machine, the minimum setup is:

```bash
git clone https://github.com/vernonthedev/nestforge.git
cd nestforge
cargo check --workspace
```

That verifies all crates and examples can be type-checked together.

# Run the example applications

The workspace includes several examples that demonstrate the main runtime styles.

```bash
cargo run -p hello-nestforge
cargo run -p hello-nestforge-graphql
cargo run -p hello-nestforge-grpc
cargo run -p hello-nestforge-microservices
cargo run -p hello-nestforge-websockets
```

Typical defaults:

- HTTP example: `http://127.0.0.1:3000`
- gRPC example: `127.0.0.1:50051`
- WebSocket default route: `/ws`
- GraphQL default route: `/graphql`

# Install the CLI

The CLI lives in `crates/nestforge-cli` and can be installed directly from the
workspace:

```bash
cargo install --path crates/nestforge-cli
```

After that you can scaffold a new project:

```bash
nestforge new demo-api
cd demo-api
cargo run
```

# Understand feature activation

NestForge is modular at the Cargo level. You only enable the feature sets your app
needs. Common examples:

```toml
nestforge = { version = "1", features = ["config", "openapi"] }
nestforge = { version = "1", features = ["graphql"] }
nestforge = { version = "1", features = ["grpc", "microservices"] }
```

This matters because many public types are re-exported conditionally from the root
`nestforge` crate.

# Recommended local verification

If you are evaluating the framework, run at least:

```bash
cargo test --workspace
cargo fmt --all --check
cargo clippy --workspace --all-targets -- -D warnings
```

That matches the contribution expectations used by the repository itself.
