# GraphQL

NestForge provides a first-class integration with **`async-graphql`**, the most popular GraphQL library in the Rust ecosystem. This allows you to combine the power of schema-first development with the elegant dependency injection of NestForge.

---

## 🚀 Getting Started

First, enable the `graphql` feature in your `Cargo.toml`.

```toml
[dependencies]
nestforge = { version = "1", features = ["graphql"] }
```

---

## 🛠️ Basic Setup

To add GraphQL to your NestForge application, you simply build your `async-graphql` schema and mount it onto the framework during bootstrap using the `with_graphql` extension.

```rust
use nestforge::{
    async_graphql::{EmptyMutation, EmptySubscription, Object, Schema},
    NestForgeFactory, NestForgeFactoryGraphQlExt,
};

// 1. Define your Query root
struct QueryRoot;

#[Object]
impl QueryRoot {
    async fn hello(&self) -> &str {
        "Hello from NestForge GraphQL!"
    }
}

// 2. Build the schema
let schema = Schema::build(QueryRoot, EmptyMutation, EmptySubscription).finish();

// 3. Bootstrap and mount
#[tokio::main]
async fn main() -> anyhow::Result<()> {
    NestForgeFactory::<AppModule>::create()?
        .with_graphql(schema) // Mounts /graphql and /graphiql
        .listen(3000)
        .await?;
    Ok(())
}
```

By default, NestForge mounts:

- **`GET /graphiql`**: The interactive playground.
- **`POST /graphql`**: The actual API endpoint.

## 💉 Dependency Injection in Resolvers

This is where NestForge truly shines. Every GraphQL request automatically carries the **NestForge Container**. This means your resolvers can easily access services registered in your modules.

Use the `resolve_graphql` helper inside your resolver methods:

```rust
use nestforge::{async_graphql::Context, resolve_graphql};
use crate::users::services::UsersService;

#[Object]
impl QueryRoot {
    async fn users(&self, ctx: &Context<'_>) -> Vec<UserDto> {
        // Resolve the service directly from the request container
        let users_service = resolve_graphql::<UsersService>(ctx)
            .expect("UsersService should be registered");

        users_service.find_all().await.unwrap_or_default()
    }
}
```

<Aside type="tip">
  By using `resolve_graphql`, you maintain a single source of truth for your
  business logic, regardless of whether it's called via REST, GraphQL, or gRPC.
</Aside>

## ⚙️ Custom Configuration

If you need to change the default paths or add custom GraphQL extensions, use the `GraphQlConfig`:

```rust
use nestforge::GraphQlConfig;

NestForgeFactory::<AppModule>::create()?
    .with_graphql_config(
        schema,
        GraphQlConfig::new()
            .with_path("/my-api")
            .with_playground_path("/play")
    )
    .listen(3000)
    .await?;
```

## 🧩 Modularity

Just like the rest of NestForge, we recommend organizing your GraphQL logic into **Feature Modules**.

```rust
// users/mod.rs
#[module(
    providers = [UsersService, UsersResolver],
    exports = [UsersService]
)]
pub struct UsersModule;
```

While NestForge doesn't currently auto-discover resolvers, the modular patterns ensure your DI remains clean and manageable as your schema grows.
