# Run the example

Example location:

```text
examples/hello-nestforge-graphql
```

Run it from the repository root:

```bash
cargo run -p hello-nestforge-graphql
```

The example exposes:

- GraphiQL at `http://127.0.0.1:3001/`
- GraphQL at `http://127.0.0.1:3001/graphql`

# Example purpose

This example shows the smallest GraphQL-first application in the workspace while still
using the NestForge module and config model.

# Files included

The example currently includes:

- `src/main.rs`
- `src/app_module.rs`
- `src/app_config.rs`
- `src/graphql/mod.rs`
- `src/graphql/schema.rs`

# Recommended reading order

Read the example in this order:

1. `src/app_module.rs`
2. `src/graphql/schema.rs`
3. `src/main.rs`

# What each file does

## `src/main.rs`

Bootstraps the app through `NestForgeFactory::<AppModule>::create()`, resolves
`AppConfig`, builds the GraphQL schema, and mounts:

- `/graphql`
- GraphiQL at `/`

It listens on port `3001`.

## `src/app_module.rs`

Registers config as a provider and exports it for downstream use. There are no HTTP
controllers because GraphQL is the primary transport.

## `src/app_config.rs`

Implements `FromEnv` and provides the default app name for the example.

## `src/graphql/schema.rs`

Defines:

- `QueryRoot`
- `MutationRoot`
- `User` and `CreateUserInput`
- an in-memory `UserStore`
- `build_schema(app_name)`

# What users learn from this example

This example teaches:

- how to mount GraphQL without traditional controllers
- how to feed app configuration into schema construction
- how to use an in-memory store for resolver examples
- how to separate module wiring from schema logic

# What is intentionally not included

This example is deliberately small. It does not try to demonstrate:

- advanced auth
- custom GraphQL context extraction from NestForge providers
- database-backed resolvers
- multi-module GraphQL composition

Use it as a transport bootstrap reference, then move to a larger application structure as
your schema grows.

# What to verify

When running the example, verify:

1. the schema is built with app config injected into it
2. `/` serves GraphiQL
3. `/graphql` serves the API
4. seeded users can be queried and created
